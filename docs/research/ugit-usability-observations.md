## 2. 易用性观察（截图层面，不杜撰实现）

### 2.1 仓库管理

观察层面：

- 主界面左侧列出已添加仓库，可快速切换
- 顶部或工具栏有"打开仓库 / 刷新 / 添加仓库"等显眼的快捷按钮
- 当前仓库的本地路径直接显示在标题或状态栏

易用性含义（这是猜测，不是截图中明示的事实）：

- 减少"打开文件对话框"次数，鼠标路径更短
- 多项目并行时不必记忆每个仓库的位置

### 2.2 变更与暂存

观察层面：

- 变更列表每行有复选框，支持批量勾选后"暂存 / 取消暂存"
- diff 视图按变更文件聚合，行级改动用绿/红高亮
- 工具栏固定位置提供"暂存选中 / 取消暂存 / 提交"等动作
- 提交信息输入框是大块多行文本框

易用性含义：

- 与"git add -i"、"git add -p"等需要记命令的方式相比，勾选 + 一个按钮更接近 Tortoise / Sourcetree 的肌肉记忆
- 提交信息大文本框鼓励规范化的 commit message

### 2.3 提交流程的细节

观察层面：

- 提交按钮和"Commit & Push"按钮分别给出，让用户选择
- 提交前给出"X files changed, Y insertions(+)"摘要
- 当工作区无实际改动时，按钮或弹层提示"空提交被禁用"

易用性含义：

- 把"是否自动推送"的选择权还给用户；不少团队规范是合本地分支、push 走 CI
- 摘要给了提交前的"最后一眼"，是错误提交的低成本防线
- 空提交禁用避免了 Git 允许空提交带来的语义噪音

### 2.4 上下文菜单的归并

观察层面：

- 一些不那么常用的功能被移到了右键菜单，例如 stash / stash pop / stash drop 这种"破坏性 + 不常做"的动作

易用性含义：

- 主界面只保留高频动作，低频动作"藏到二级菜单"，避免主界面被塞满

### 2.5 横向与竖向条幅

观察层面：

- 第 4、5 张是窄长截图，比例说明可能是菜单、状态条或 Toast 这类辅助 UI

易用性含义：

- 不抢主界面焦点的次要信息（操作成功、网络状态、锁提示）走这种条幅

## 3. 对 xgit 的借鉴清单（属于"做到什么程度"，不是"必须照抄"）

任何具体取舍都需等首版可工作之后用真实用户验证。下面是先列入 Backlog 的条目，原文存档备查。

| UGit 截图观察 | xgit 是否要做 | 备注 |
|---|---|---|
| 左侧仓库列表 + 添加按钮 | 是 | 多仓库是游戏工作室常态 |
| 暂存复选框 | 是 | CLI `git add -p` 难，对非程序员友好 |
| 大 commit 文本框 + 提交摘要 | 是 | 鼓励规范 commit message |
| Commit 与 Commit & Push 分别按钮 | 是 | 各团队 push 节奏不同 |
| 空提交拦截 | 是 | 几乎零成本 |
| stash 移入右键菜单 | 是 | 防误操作 |
| 横向 / 竖向通知条 | 是 | 锁状态、fetch 进度用得上 |

## 4. 不要照抄的地方

- UGit 是 Electron + TypeScript + Monaco + LuckySheet，包体积和内存成本明显（参考 ADR-0001 §3.1"系统资源消耗低"）。这些 UI 框架不能因为"截图看着顺眼"就引进 Wails + Go 路线
- UGit 的工蜂集成、SVN/P4 互操作、Excel Diff & Merge 与 xgit 范围无关，不在 v1.0 视野
- UGit 是闭源且深度绑定腾讯内部平台，公开渠道拿不到 commit / fetch 性能的可复现数据；不要把"看起来很快"当作性能结论
